/* ---------------------------------------------------------------------
   staff_institute_roles
   Maps a staff member to an institute they manage, with a role. Read by the
   leave-approval routing (section 2 below). A staff can manage many
   institutes and hold several roles. Business rule (enforced in app logic,
   not by a constraint): within one institute at most ONE moderator and ONE
   approval; admin may be many.
   --------------------------------------------------------------------- */
CREATE TABLE `staff_institute_roles` (
  `id` int unsigned NOT NULL AUTO_INCREMENT,
  `staff_id` varchar(100) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci NOT NULL,
  `institute_id` varchar(100) CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci NOT NULL,
  `role` enum('admin','moderator','approval') CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci NOT NULL,
  `created_at` timestamp NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at` timestamp NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`) USING BTREE,
  UNIQUE KEY `uq_staff_institute_role` (`staff_id`,`institute_id`,`role`) USING BTREE,
  KEY `staff_id` (`staff_id`) USING BTREE,
  KEY `institute_id` (`institute_id`) USING BTREE,
  KEY `role` (`role`) USING BTREE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci;


/* =====================================================================
   PROCESS FLOW â€” LEAVE APPROVAL ROUTING
   How a leave request auto-populates `leave_approval` (who must approve).
   =====================================================================

   ---------------------------------------------------------------------
   1. STAFF <-> INSTITUTE MANAGEMENT (prerequisite)
   ---------------------------------------------------------------------
   - One staff can MANAGE many institutes (many-to-many).
   - For each managed institute the staff holds one or more roles:
         admin | moderator | approval
   - Rule: within ONE institute there is at most ONE `moderator` and at most
     ONE `approval` (admin may be many).
   - Institutes are hierarchical; each institute has a `level_type`.
     SMALLER level_type = HIGHER authority  (10 outranks 40 outranks 50).
   - This mapping is the `staff_institute_roles` table defined above; it is
     read by the routing flow below.
   - CRUD for it is built (API: /api/staff-institute-roles; Vue: "Manage
     Roles"). The routing flow in sections 2 and 4 is BUILT in
     services/leaveflow + handlers/leave.go.

   ---------------------------------------------------------------------
   2. ON CREATE LEAVE -> RESOLVE APPROVER  ("when request live => auto")
   ---------------------------------------------------------------------
   Step 1. Match a `leave_roles` row by:
              leave_type  +  requested days in [min_duration, max_duration]
   Step 2. Read `approve_level` from that row.
              approve_level  ==  `level_type` in `institutes`.
   Step 3. Resolve the requester's institute context:
              read requesting staff's `institute_ids` (institute chain), then
              load the `level_type` of each of those institutes.
   Step 4. Walk tiers from the start level toward HIGHER authority
           (DECREASING level_type number), tier by tier, down to 10:

              for L in (start_level, next-higher tier, ... , 10):
                  a. find staff MANAGING an institute at level_type L
                     with role = 'moderator'
                        -> if found: INSERT into leave_approval, STOP.
                  b. else find staff managing at level_type L
                     with role = 'approval'
                        -> if found: INSERT into leave_approval, STOP.
                  c. else: go to the next-higher tier (smaller level_type).

           Stop on the first manager found, or after level_type = 10.

   ---------------------------------------------------------------------
   EXAMPLE
   ---------------------------------------------------------------------
   leave_roles gives approve_level = 50, requesting staff level_type = 40:
     - search level_type 40 for a 'moderator'; if none, an 'approval'
     - if level 40 has no manager -> 30 -> 20 -> 10
     - first match found is inserted into leave_approval; stop at 10.
   (The walk starts at the requester's own level_type, 40.)

   ---------------------------------------------------------------------
   3. ON FINAL APPROVE -> leave_year
   ---------------------------------------------------------------------
   Runs ONCE, when the request becomes fully approved (end of section 4).
   Upsert `leave_year` (staff_id, leave_type_id, l_year): accumulate
   total_day, set max_days / is_reset from `leave_type`,
   leave_remaining = max_days - total_day.

   ---------------------------------------------------------------------
   4. APPROVER TASK WORKFLOW â€” process a pending step   (NOT built yet)
   ---------------------------------------------------------------------
   Each `leave_approval` row is a TASK for the staff named in it. The flow is
   multi-step: one approver acts, then the chain either ends or spawns the
   next approver's task. Sections 2 and 4 share the SAME walk (section 2 step 4).

   TASK LIST (per approver)
      On login an approver sees every request awaiting them =
         leave_approval WHERE staff_id = <current user> AND status = 'pending'
      joined to `leave` for the request details.
      (Already queryable: GET /leave-approvals?staff_id=<me>&status=pending;
       the act-on-task logic below is what still needs building.)

   REJECT  (one rejection kills the whole request)
      1. this leave_approval row -> status='rejected', approved_at=now
      2. MAIN `leave` -> status='rejected', reject_reason=<reason>,
         approved_by=<this approver> (who), approved_at=now (when)
      3. STOP â€” no further approvers.

   APPROVE  (advance the chain)
      1. this leave_approval row -> status='approved', approved_at=now
      2. Is the chain COMPLETE?
           COMPLETE  when   current approver's role = 'approval'
                     AND    requester.level_type >= current approver.level_type
                            (approver at EQUAL-OR-HIGHER authority than the
                             requester; smaller level_type = higher).
           -> if COMPLETE: MAIN `leave` -> status='approved',
              approved_by=<this approver>, approved_at=now, then run the
              leave_year upsert (section 3). DONE.
      3. else NOT complete -> find the NEXT approver and INSERT a new pending
         leave_approval row, using the SAME walk as section 2 but restricted to
         role = 'approval':
           - look for a role='approval' manager at the current tier, then the
             next-smaller EXISTING level_type, ... up to level_type 10.
           - insert the pending row (l_level / approve_level = next step), STOP.
         That approver then sees the task; the cycle repeats from TASK LIST.

   EXAMPLE  (requester level_type = 40)
      - section 2 routing picked a moderator at 40 -> pending task.
      - moderator(40) approves -> role != 'approval' -> find next 'approval':
        e.g. approval at 30 -> insert pending task for them.
      - approval(30) approves -> role='approval' AND 40 >= 30 -> COMPLETE ->
        MAIN leave = approved, leave_year upserted.

   STATUS: BUILT.
     POST /leave-approvals/:id/approve  -> approve a step (advance/complete)
     POST /leave-approvals/:id/reject   -> reject the whole request
   Routing lives in services/leaveflow. (The one-shot POST /leaves/:id/
   approve|reject were removed â€” admins are view-only for leave.)
   AUTHZ: the caller must be the assigned approver (users.staff_id == the
   leave_approval row's staff_id). ADMINS ARE VIEW-ONLY and get 403 if they try
   to approve/reject; they can still view requests and the approval timeline
   (frontend: /leaves/:id, views/leaves/LeaveDetail.vue).
   BREAK-GLASS: admin-only POST /leaves/:id/override-approve|override-reject
   force-resolve a STUCK request (flip pending steps, finalize, approve =>
   upsert leave_year). Audited by MERGING an "override" key into
   leave.approve_document (other keys preserved): {...existing,
   override:{by,action,note,at}}.
   ===================================================================== */
